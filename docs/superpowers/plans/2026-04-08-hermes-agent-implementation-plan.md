# Hermes Agent 专题页实施计划

## 1. 目标

基于已确认的设计文档，实现一个新的 `hermes-agent` 专题页，用站点现有专题页体系完成以下目标：

- 先建立 Hermes Agent 的正确认知
- 再解释其核心能力与差异
- 最后承接官方文档与行动入口

本计划默认该页面将作为站内新的静态专题页实现，并复用现有 `series-page` 体系与通用组件。

## 2. 预期产出

本次实现完成后，应至少包含：

- 新页面路由：`/hermes-agent`
- 该页面对应的 SEO 配置
- 页面专用的数据配置或内容常量
- 可复用的模块化页面结构
- 指向 Hermes Agent 公开资料的外部入口
- 基础响应式适配

如果实现过程中发现 Hermes 页的某些模块适合复用到后续专题页，可以顺手抽出低风险公共结构，但不做与当前需求无关的大范围重构。

## 3. 实施原则

- 优先复用现有专题页骨架与样式能力
- 文案表达贴近设计 spec，但事实表述必须对齐公开资料
- 页面重点是认知建立，不追求堆叠信息量
- 避免把外部资料大段搬运到页面中
- 不新增复杂数据流、远程请求或运行时依赖

## 4. 实现范围

### 4.1 在范围内

- 新增 Hermes Agent 专题页
- 新增支撑该页的数据配置
- 补充必要的导航/入口曝光
- 补充 SEO 和结构化信息
- 页面静态验证与响应式检查

### 4.2 不在范围内

- 搭建 Hermes Agent 文档镜像
- 自动同步外部官方内容
- 增加复杂交互动画或浏览器端状态逻辑
- 新建一整套 Hermes 独立设计系统
- 对全站专题体系做系统性重构

## 5. 推荐实现顺序

### 阶段 1：确定页面落点与复用策略

目标：确认页面放置方式，并减少后续返工。

任务：

1. 确认页面文件位置，建议新增：
   - `app/pages/hermes-agent.vue`
2. 确认复用骨架：
   - 以现有 `series-page` / `series-hero` / `series-panel` 为主
3. 确认需要的数据文件：
   - 建议新增 `app/data/hermes-agent.ts`
4. 检查是否需要在以下位置增加入口：
   - 首页相关专题入口
   - `topics`、`ecosystem`、`secondary-development` 或其他最合适的专题承接页

完成标准：

- 页面文件路径和数据文件路径确定
- 确认无需为本页单独创建全新布局体系

### 阶段 2：搭建数据与文案结构

目标：先把页面所需内容对象化，避免在 `.vue` 文件内堆长数组和大段文案。

任务：

1. 在 `app/data/hermes-agent.ts` 中定义页面所需数据结构，建议按模块拆分：
   - `hero`
   - `coreBeliefs`
   - `capabilityFlow`
   - `scenarios`
   - `resourceLinks`
   - `faqItems`
2. 为每个数据项保留足够清晰的字段，避免模板层做过多字符串拼接
3. 对外部链接统一记录：
   - 标题
   - 描述
   - href
   - 类型或标签
4. 文案统一按设计 spec 收口，确保：
   - 不把 Hermes 写成人格设定页
   - 不写成安装文档摘要
   - 不与官方定位冲突

完成标准：

- 页面内容在数据文件中可读、可维护
- `.vue` 文件主要负责展示，不承担大段内容编排

### 阶段 3：实现页面结构

目标：把设计 spec 中的 6 个模块完整落地。

任务：

1. 实现 Hero 首屏
   - 标题、副标题、2 个主动作、1 个次级外链
   - 右侧说明卡，解释“这一页帮你看清什么”
2. 实现核心认知卡片区
   - 4 张卡片
   - 每张卡片一句判断 + 一段解释
3. 实现能力主轴区
   - 用连续步骤或流程卡片表达 4 到 5 步能力链
4. 实现典型场景区
   - 控制在 2 到 3 个场景
5. 实现文档与行动入口区
   - 按“下一步想做什么”组织，不是平铺链接
6. 实现 FAQ / 边界区
   - 解决常见认知偏差

完成标准：

- 页面结构与 spec 一一对应
- 页面阅读顺序自然，不出现信息堆叠和模块争抢

### 阶段 4：视觉收口与响应式适配

目标：让页面既融入站点，又有 Hermes 专题的辨识度。

任务：

1. 在 scoped CSS 中补充页面局部样式
2. 尽量复用现有系列页 class 命名和 spacing 节奏
3. 仅在必要处新增局部视觉元素：
   - Hero 状态标签
   - 能力链样式
   - 场景卡片层级
4. 检查移动端排版：
   - 标题换行
   - 卡片堆叠
   - 外链按钮宽度与可点击性

完成标准：

- 桌面端与移动端都能顺畅阅读
- 页面视觉不脱离站点既有风格

### 阶段 5：SEO 与站点接入

目标：让该专题页成为真正可发现、可索引、可承接的站点资产。

任务：

1. 配置 `useSeo`
   - 标题
   - 描述
   - path
   - type
2. 视现有站点模式决定是否补充：
   - `schemaType`
   - item list / related links 等结构化信息
3. 确认页面是否被现有 sitemap / prerender 策略覆盖
4. 选择至少一个合适的站内入口，把用户导流到该页

完成标准：

- 页面具备完整基础 SEO
- 页面不是“孤立路由”

### 阶段 6：验证与收尾

目标：确保页面在视觉、信息与链接层面可交付。

任务：

1. 本地运行页面并检查：
   - 无编译错误
   - 页面能正常访问
   - 外链无明显错误
2. 手动检查核心断点：
   - 桌面宽屏
   - 常规笔记本宽度
   - 手机宽度
3. 检查文案一致性：
   - 是否统一使用 “Hermes Agent”
   - 是否存在“人格化角色页”倾向
   - 是否存在事实表达过度推断
4. 检查站内入口与回流：
   - 从入口页能否进入 Hermes 页
   - 从 Hermes 页能否顺畅跳到官方资源

完成标准：

- 页面在功能和表达上达到 spec 要求
- 主要入口与出口都成立

## 7. 内容映射建议

建议将页面内容与公开资料映射如下：

- Hero 与核心认知：
  - 基于官方首页与 Features Overview 的定位表达
- 能力主轴：
  - 基于 Features Overview 的能力类别做中文化重组
- 文档入口：
  - 对接 Quickstart、Features Overview、Security、Docs / GitHub
- FAQ：
  - 用公开资料能支持的判断做纠偏，不写未经证实的深层能力结论

## 8. 风险与注意点

### 风险 1：页面重新滑回“人格化角色页”

应对：

- 标题和首屏定义必须把 Hermes 放回“自治 agent 系统”语境
- 任何“高执行力”表达都只能作为气质层，而不是事实定义

### 风险 2：页面做成外链导航站

应对：

- 文档链接只放在后段
- 中段必须先完成完整认知叙事

### 风险 3：模块过多导致专题页失焦

应对：

- 严格限制模块数
- 场景控制在 2 到 3 个
- 不加额外长目录或过度 FAQ

### 风险 4：入口过多导致站点信息架构变乱

应对：

- 先选一个主入口承接
- 如需更多入口，后续再基于数据判断是否扩展

## 9. 验收清单

上线前至少满足以下检查项：

- 存在可访问的 `/hermes-agent` 页面
- 页面标题与副标题能清楚解释 Hermes 是什么
- 页面中段明确展示能力链，而不是散乱功能卡片
- 页面有 2 到 3 个场景证明 Hermes 的适用性
- 页面有明确的 Quickstart / Features / Security / GitHub 承接
- 页面在移动端仍可读、CTA 可点击
- 页面文案未偏离官方事实
- 页面已接入至少一个合理的站内入口

## 10. 建议执行策略

建议按两个提交完成：

1. 第一提交：页面数据、页面结构、基础样式
2. 第二提交：站内入口、SEO、响应式细节与校对

这样可以降低一次性改动过大的风险，也更方便 review。
