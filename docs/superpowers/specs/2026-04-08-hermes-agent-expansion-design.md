# Hermes Agent 专题扩充设计文档

## 1. 背景

当前站点已经有一个 `Hermes Agent` 总览页，用来帮助用户快速建立基础认知：

- Hermes Agent 是什么
- 为什么它不同于普通聊天 agent
- 用户下一步应该去哪些官方资料入口

这个版本已经适合做“第一认知页”，但内容体量仍然偏轻，无法承接用户继续深入理解的需求。随着页面被放到站点导航与专题入口中，单页结构开始暴露两个问题：

- 如果继续把内容往同一页里堆，页面会迅速变成一张又厚又散的长页
- 如果不扩展分页，Hermes 的产品理解、运行方式、关键机制和安全边界都只能停留在浅层介绍

因此，这次需求不是“单纯加长总览页”，而是把现有 Hermes 页升级成一组有层级的专题体系。

## 2. 设计目标

### 2.1 主目标

将 Hermes 从单页认知专题扩展为：

**一个以总览页为中心、配套 5 个深读分页的专题体系。**

它需要同时覆盖三类内容：

- 产品理解型
- 使用落地型
- 深度机制型

但整体表达应以“专题解读”优先，而不是复制官方文档目录。

### 2.2 次目标

- 保持现有 Hermes 总览页的旗舰专题定位
- 通过分页把复杂内容按问题边界拆开
- 形成站内可持续扩展的专题中心结构
- 为后续 SEO、内链和相关专题回流提供更丰富的落点

### 2.3 非目标

- 不把这一组页面做成官方文档镜像
- 不把每个分页都写成安装教程或参数手册
- 不引入复杂的 docs 式树形侧边栏
- 不把 Hermes 专题做成独立子站或独立设计系统

## 3. 定位结论

Hermes 扩充后的形态定义为：

**专题中心 + 深读分页**

而不是：

- 单页超长专题
- 纯文档中心
- 站内复制版官方 docs

这组页面要保留 Hermes 总览页的“旗舰专题感”，同时让每个分页各自回答一个明确问题。

## 4. 推荐结构

### 4.1 页面总数

推荐扩展为 6 个页面：

1. `/hermes-agent`
2. `/hermes-agent/capabilities`
3. `/hermes-agent/operations`
4. `/hermes-agent/memory-and-skills`
5. `/hermes-agent/automation-and-surfaces`
6. `/hermes-agent/security`

### 4.2 结构原则

- 总览页负责建立认知和分流
- 分页负责把某一个问题讲透
- 任一分页都不再承担“总览页缩写版”的职责
- 页面之间应强回链，但不形成复杂层级导航负担

## 5. 信息架构

### 5.1 总览页

路径：

- `/hermes-agent`

职责：

- 定义 Hermes Agent 是什么
- 解释 Hermes 与普通聊天 agent、一次性 workflow、单轮 copilot 的差别
- 提供整组 Hermes 专题的阅读地图
- 引导用户进入更适合自己的分页

总览页新增内容建议：

- 专题地图卡片
- “你现在更适合先看哪一页”分流模块
- 2 到 3 个高信号对比模块
- 各分页的摘要卡片

总览页不应继续承载：

- 过细的机制拆解
- 过细的运行步骤
- 过细的安全细则

### 5.2 能力地图页

路径：

- `/hermes-agent/capabilities`

核心问题：

> 为什么 Hermes 能被理解成一个长期工作的自治 agent 系统？

职责：

- 用能力层解释 Hermes 成立的原因
- 讲清长期运行、记忆、工具、自动推进、多入口协作如何构成整体系统

建议模块：

- 能力总图
- 长期运行能力
- 记忆与状态能力
- 工具与执行能力
- 自动推进能力
- 多入口协作能力
- 能力之间的支撑关系

边界：

- 这页解释“能力地图”
- 不承担“怎么安装”和“安全细则”职责

### 5.3 运行与使用页

路径：

- `/hermes-agent/operations`

核心问题：

> 真正开始使用 Hermes 时，应该怎么理解它的运行方式？

职责：

- 解释 Hermes 的运行形态
- 解释用户该从哪种使用入口开始
- 解释长期运行时最需要关注的事项

建议模块：

- Hermes 的典型运行形态
- 从 Quickstart 到长期运行的理解路径
- 任务来源与入口
- CLI、消息入口、长期运行各适合什么
- 长期运行时的观察重点
- 常见使用误区

边界：

- 这页是“使用理解页”
- 不是完整安装教程

### 5.4 记忆、技能与子 agent 页

路径：

- `/hermes-agent/memory-and-skills`

核心问题：

> 为什么 Hermes 会随着运行时间变强？

职责：

- 解释长期记忆、技能积累和子 agent 协作在 Hermes 中扮演的角色
- 强化 Hermes 与一次性 agent 的深层差异

建议模块：

- 记忆为什么重要
- 技能在 Hermes 中的角色
- 子 agent 为什么成立
- 三者如何一起形成长期积累
- 最适合体现这套能力的任务
- 不适合把这套能力开太重的场景

边界：

- 这页聚焦“长期积累机制”
- 不承担入口与自动化主讲职责

### 5.5 自动化、浏览器与入口页

路径：

- `/hermes-agent/automation-and-surfaces`

核心问题：

> Hermes 是怎么接到任务，并接触真实世界的？

职责：

- 解释 CLI、消息入口、浏览器、自动化任务等外部接面
- 解释这些入口为什么会改变 agent 的工作方式

建议模块：

- CLI 与交互式入口
- 消息与外部入口
- 浏览器控制
- 自动化与定时任务
- 长期任务如何被触发和回报
- 这些入口与表面的差异

边界：

- 这页聚焦“表面与触发面”
- 不承担能力全景和安全边界总讲职责

### 5.6 安全与边界页

路径：

- `/hermes-agent/security`

核心问题：

> 一个长期在线、会行动的 agent，最值得警惕的边界是什么？

职责：

- 帮用户建立 Hermes 的风险判断
- 解释长期运行的权限与安全问题为何不能当附录处理

建议模块：

- 为什么 Hermes 的安全问题不是附录
- 长期运行风险与普通聊天 agent 的区别
- 权限、工具、浏览器、自动化的风险面
- 更稳的启用顺序
- 哪些能力应该最后再打开
- 官方安全资料阅读建议

边界：

- 这是“安全认知页”
- 不是逐条参数手册

## 6. 页面间关系

### 6.1 组织方式

整组 Hermes 页面应表现为：

- 1 个专题中心
- 5 个深读分页

阅读感应更接近“有中心页的专题组”，而不是文档树。

### 6.2 导航方式

不建议为 Hermes 引入完整 sidebar。

推荐使用三层承接：

- 总览页专题地图
- 各分页顶部的同系列导航
- 各分页底部的“继续阅读 / 返回总览”

### 6.3 页面回链规则

每个分页都应清楚回答：

- 我在这组 Hermes 专题里负责什么
- 我前面建议先看哪一页
- 我后面建议继续去哪一页

### 6.4 主导航策略

- 主导航继续只暴露 `/hermes-agent`
- 不在全站主导航中直接展开全部 Hermes 子页
- Hermes 子页主要通过总览页、分页互链和站内专题推荐卡进入

## 7. 视觉与内容风格

### 7.1 总体风格

整组 Hermes 页面继续沿用当前站点专题体系：

- `series-page` 语系优先
- 卡片化、判断句式、专题页阅读节奏
- 纸面质感与现有主色系统保持一致

### 7.2 页面表达差异

- 总览页偏“建立认知”
- 分页偏“讲透一层”
- 所有分页依然是专题页，不切换成纯文档模板

### 7.3 文案口径

本组页面采用：

**专题解读优先，使用承接为辅**

这意味着：

- 先回答“它是什么 / 为什么成立 / 什么时候适合”
- 再承接“我下一步该怎么用 / 去看哪里”

避免偏差：

- 避免全页参数手册化
- 避免大段照搬官方英文说明
- 避免把每个分页都写成独立总览页

## 8. 推荐实现顺序

如果后续进入实现，推荐顺序如下：

1. 先重构总览页为专题中心
2. 先实现 `capabilities` 与 `operations`
3. 再实现 `memory-and-skills`
4. 再实现 `automation-and-surfaces`
5. 最后实现 `security`

原因：

- `capabilities` 与 `operations` 是整组专题最基础的两层
- `memory-and-skills` 是 Hermes 差异化最强的一层
- `automation-and-surfaces` 与 `security` 更适合作为后续深读页承接

## 9. 风险与约束

### 风险 1：做成“站内复制版官方文档”

应对：

- 每页先讲判断，再讲入口
- 页面职责以“中文解读与分层”为主，而非逐节翻译

### 风险 2：总览页继续过厚

应对：

- 明确把机制、运行和安全细节下沉到分页
- 总览页只保留认知、分流和高层对比

### 风险 3：分页之间再次重叠

应对：

- 每页只回答一个核心问题
- 在 spec 中明确边界，避免实现阶段随意加料

### 风险 4：导航层级过重

应对：

- 不新增复杂 sidebar
- 只保留专题中心式导航

## 10. 成功标准

当这组 Hermes 专题扩充完成后，应满足：

- 用户能从 `/hermes-agent` 快速看懂整组内容结构
- 每个分页都承担明确职责，不互相抢内容
- 用户能沿分页顺序逐层加深理解，而不是被迫看一张超长单页
- Hermes 专题整体仍然保持站点专题页风格，而不是突变成 docs 镜像
- 总览页继续承担旗舰专题页职责

## 11. 实施边界

本 spec 只定义 Hermes 专题扩充后的：

- 页面结构
- 分页职责
- 导航策略
- 内容风格

后续实现阶段再单独规划：

- 路由文件如何组织
- 数据文件如何拆分
- 页面模块如何复用
- 总览页与分页的具体改写顺序
- SEO 与内链策略
