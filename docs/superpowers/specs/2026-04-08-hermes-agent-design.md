# Hermes Agent 专题页设计文档

## 1. 背景

当前站点已经形成了较稳定的专题页表达方式，尤其是 `/tools`、`/ecosystem`、`/secondary-development` 这类页面，已经不再只是“信息列表”，而是承担“帮助用户建立认知、理解边界、再进入下一步内容”的内容产品页。

本次新增的 `hermes-agent` 专题页，需要延续这种站点风格，但同时要避免两个偏差：

- 不能把 Hermes Agent 写成一个纯人格化角色页，因为公开资料显示它本质上是一个长期在线、可积累、可扩展的自治 agent 系统
- 也不能退化成一个普通文档索引页，因为用户希望它承担“旗舰专题页”的任务，先建立清晰认知，再承接文档和后续动作

因此，这一页的设计目标不是“把 Hermes 讲得更酷”，而是把 Hermes Agent 的真实定位讲得更清楚、更有记忆点。

## 2. 设计目标

### 2.1 主目标

让已经了解 OpenClaw 的用户，在进入本页后快速建立对 Hermes Agent 的正确认知：

- Hermes Agent 是一个长期在线、会积累记忆与技能、能持续推进任务的自治 agent 系统
- 它不同于普通聊天 agent、一次性 workflow 或单次 copilot
- 用户看完这一页后，知道它适合什么、为什么成立、下一步该看哪里

### 2.2 次目标

- 把 Hermes Agent 作为站内的一个旗舰专题表达出来，提升页面辨识度
- 为后续的快速开始、功能文档、安全说明和 GitHub 入口提供统一承接
- 保持与站点现有专题页的视觉和信息架构一致性，避免风格漂移

### 2.3 非目标

- 不把本页做成完整安装教程
- 不把本页做成纯 FAQ 或纯文档目录
- 不构建纯拟人化的角色设定页
- 不在本页完整展开所有技术细节，避免与官方文档职责重复

## 3. 目标用户

本页主要面向：

- 已经对 OpenClaw 有基本了解的用户
- 想进一步理解 Hermes Agent 是什么、与常见 agent 形态有何不同的用户
- 希望在进入快速开始或功能文档前先建立整体判断的用户

本页不优先服务完全陌生的新用户，也不优先服务需要深度技术细节的开发者；这两类诉求分别由首页/入门页与官方文档承接。

## 4. 定位结论

`hermes-agent` 页面定义为：

**一个基于官方事实的旗舰能力专题页。**

它的核心任务是先建立认知，再承接文档与行动入口。

页面表达上，保留一定“旗舰感”和“高执行力”的气质，但叙事必须统一到 Hermes Agent 的真实产品定位：

- 长期在线
- 可持续积累
- 能主动推进任务
- 可通过多种入口接入
- 具有自治 agent 系统属性

## 5. 信息架构

### 5.1 页面骨架

页面采用 `series-page` 体系作为主骨架，而不是 `collection-page`。

原因：

- `series-page` 更适合“先建立整体认知，再分块解释能力，再给出下一步入口”的专题结构
- `collection-page` 更适合长篇正文、深度导读和页内导航，不适合作为 Hermes Agent 的第一认知页
- Hermes Agent 需要的是“能力专题页”，不是“教程正文页”

### 5.2 页面顺序

页面按以下顺序组织：

1. Hero 首屏
2. 核心认知卡片
3. 能力主轴区
4. 典型场景区
5. 文档与行动入口区
6. FAQ / 认知边界区

这一顺序的原则是：

- 先定义 Hermes 是什么
- 再解释它为什么和普通 agent 不一样
- 再用能力链和场景证明这个定义
- 最后再导向具体文档和下一步动作

## 6. 模块设计

### 6.1 Hero 首屏

#### 目标

在第一屏内直接完成“身份定义”和“差异建立”。

#### 内容要求

- 标题直接定义 Hermes Agent 的身份
- 副标题解释其与普通聊天 agent、一次性 workflow、单次 copilot 的差别
- 主 CTA 只保留 2 个，避免注意力分散
- 右侧或侧栏区域不堆数据，而是说明“这一页帮你看清什么”

#### 推荐标题方向

- `Hermes Agent：长期在线、会积累、能推进任务的自治 Agent`
- `Hermes Agent：一个会持续成长的高执行力 Agent`
- `Hermes Agent：不只是聊天，而是能持续工作的 Agent 系统`

#### 推荐 CTA

- `查看快速开始`
- `浏览核心能力`

GitHub 入口如果需要展示，应作为次级链接，不与主 CTA 同权。

### 6.2 核心认知卡片

#### 目标

用最短的信息密度，帮助用户建立 Hermes Agent 的四个基本判断。

#### 推荐卡片主题

- `长期运行`
  - 解释 Hermes 不是一次性对话，而是适合长期在线运行的 agent
- `记忆与成长`
  - 解释它会积累记忆、技能或经验，能力不是完全静态的
- `自动推进`
  - 解释它不仅回答问题，还可以持续推进任务
- `多入口协作`
  - 解释它可以通过 CLI、消息、自动化等入口接入，而不是局限在单一界面

#### 设计原则

- 每张卡片先给一句判断，再给一段解释
- 不使用过度技术术语
- 不直接抄写官方 feature 列表，而是做认知层转述

### 6.3 能力主轴区

#### 目标

把 Hermes Agent 解释成一条连续工作的能力链，而不是一组零散 feature。

#### 推荐组织方式

建议使用 4 到 5 段连续流程：

1. `接收任务`
2. `记住上下文`
3. `调用技能与工具`
4. `并行或异步推进`
5. `持续回报结果`

#### 设计原则

- 用“链路感”强化 Hermes 的自治 agent 属性
- 避免简单平铺功能卡片
- 每一步都要指向“为什么它能长期工作”

### 6.4 典型场景区

#### 目标

用少量高代表性场景证明 Hermes 的能力主张成立。

#### 推荐场景方向

- 长期研究与跟踪
- 自动巡检与定时处理
- 多入口消息驱动的工作助手

#### 场景卡片模板

每个场景统一包含：

- 场景名称
- Hermes 在这里扮演什么角色
- 为什么普通 agent 不一定能做好

#### 设计原则

- 场景只做证明，不抢主叙事
- 不扩展成案例库
- 数量控制在 2 到 3 个

### 6.5 文档与行动入口区

#### 目标

在认知建立后，为用户提供清晰、低犹豫的下一步动作。

#### 组织方式

入口不按“链接类型”组织，而按“用户下一步想做什么”组织。

推荐入口：

- `我想先跑起来` -> Quickstart
- `我想看核心能力` -> Features Overview
- `我想确认安全边界` -> Security
- `我想继续深入` -> Docs / GitHub

#### 设计原则

- 入口要像“下一步建议”，而不是冷链接列表
- 维持专题页的承接感

### 6.6 FAQ / 认知边界区

#### 目标

解决用户最常见的理解偏差，帮助页面收口。

#### 推荐问题方向

- Hermes 和普通聊天 agent 有什么不同
- Hermes 更适合哪些长期任务
- Hermes 是 workflow 工具还是自治 agent
- 开始使用前应该先理解什么

#### 设计原则

- 问题以“认知纠偏”为主
- 不扩展成技术支持页

## 7. 文案与语气

### 7.1 语气方向

文案整体延续站点现有风格：

- 克制
- 清晰
- 中文信息密度高
- 偏判断句和解释句

但 Hermes 页需要额外加强“推进感”和“执行感”，因此更适合使用以下语言倾向：

- 持续运行
- 主动推进
- 并行处理
- 长期接棒
- 持续积累

### 7.2 禁止偏差

- 避免把 Hermes 写成纯人格化角色
- 避免夸张营销文案
- 避免堆砌零散技术名词
- 避免把页面写成安装教程摘要

## 8. 视觉方向

### 8.1 视觉原则

- 延续站点当前的纸面质感、青蓝与暖棕主色、卡片式信息结构
- 不为 Hermes 另起一套完全独立的品牌系统
- 在 Hero 和能力主轴区适度强化“操作感”和“状态感”

### 8.2 视觉表现建议

- Hero 区强调标题、状态标签、能力判断，而不是大篇正文
- 能力主轴区更适合带流程带、步骤卡片或连续结构
- 场景区视觉上应弱于 Hero 和能力主轴区
- 文档入口区保持清晰、安定、可点击，不做过度装饰

## 9. 内容来源与事实边界

本页涉及的 Hermes Agent 定位与能力描述，应统一参考公开可核实资料，尤其是：

- 官方首页
- Quickstart
- Features Overview
- Security
- GitHub 仓库说明

本页允许对内容做中文化和认知层转述，但不应虚构官方未表达的产品定位。

## 10. 成功标准

当页面完成后，应满足以下标准：

- 用户在第一屏即可理解 Hermes Agent 不是普通聊天 agent
- 用户能在页面中段理解 Hermes 的长期运行、记忆积累和任务推进逻辑
- 用户能在页面末尾自然进入官方文档、快速开始或 GitHub
- 页面整体风格与站点现有专题体系一致
- 页面不会被理解成“人格化角色页”或“简单文档导航页”

## 11. 实施边界

本设计文档只定义专题页的信息架构、内容表达和设计方向。

后续实现阶段需要单独规划：

- 页面路由与命名
- 是否需要新增数据文件
- 文案落地与来源映射
- SEO 文案和结构化数据
- 响应式细节与组件复用策略

这些内容不在本 spec 中展开，由后续 implementation plan 承接。
