# 内容结构与系统架构

如果你只把 OpenClaw 看成一个聊天入口，很快就会在接渠道、排错、权限控制和长期运行时遇到理解断层。官方文档把它拆成 Onboarding、Channels、Tools、Models、Platforms、Gateway & Ops、Reference 等模块，本质上是在提醒你：这是一套分层系统，而不是一个单点应用。

## 一张心智图先看懂

可以先把 OpenClaw 理解成五层：

1. 入口层：Channels 与 Web / UI 入口
2. 控制层：Control UI、Onboarding、配置与诊断
3. 能力层：Agents、Tools、Models、Hooks
4. 运行层：Platforms、Gateway、远程访问、认证、日志
5. 内容与运营层：文档、版本跟踪、社区反馈、最佳实践

## 1. 入口层：用户如何接近系统

入口层负责“从哪里进入 OpenClaw”。它不只是一种网页界面，还可能是：

- 一个网页控制入口
- 一个聊天或消息入口
- 一个团队协作入口
- 一个供运维和管理员使用的管理入口

入口层决定的是交互方式，不决定核心能力本身。

## 2. 控制层：让系统可见、可查、可改

这层最典型的内容就是官方单独强调的 Onboarding 与 Control UI。它们的作用不是“更漂亮地聊天”，而是：

- 帮你完成第一轮初始化
- 让你知道系统当前是否健康
- 把配置、状态、认证和可用能力可视化
- 在出问题时提供检查入口

对第一次使用来说，控制层的价值通常高于多渠道。

## 3. 能力层：真正决定 OpenClaw 能做什么

能力层通常由以下几个部分组成：

### Agents

决定代理的角色、行为边界和默认工作方式。

### Tools

决定代理能调用哪些外部能力，例如命令执行、浏览器、外部 API 或业务工具。

### Models

决定推理能力来自哪里，以及不同场景下如何选择模型与回退策略。

### Hooks

决定在请求前后、工具执行前后、记录日志或注入规则时，系统怎样扩展行为。

## 4. 运行层：长期运行时必须正视的边界

当你从“试用”进入“长期运行”阶段，问题会从功能转向运维：

- 入口是否暴露到公网
- 认证和授权是否打开
- 渠道凭证如何管理
- 日志、诊断和更新如何处理
- 远程运行时谁可以访问管理界面

这也是为什么官方把 Gateway & Ops 单独放出来。很多事故并不是模型能力不足，而是运行边界没有管好。

## 5. 内容层：中文站为什么要这样组织

从中文站角度，我们把文档拆成入门、安装、架构、渠道、Control UI、Hooks、安全、排错、迁移和更新跟踪，目的是让用户按问题而不是按源码目录来理解系统。

对应关系大致是：

- 想知道“它是什么”：看产品和适用人群文档
- 想知道“第一次怎么开始”：看快速入门、安装、onboarding
- 想知道“各模块怎么分工”：看架构、Control UI、渠道、Hooks
- 想知道“长期怎么跑”：看安全、排错、版本迁移、更新跟踪

## 最容易搞混的三个边界

### Control UI 不等于主要使用入口

它更像系统管理和观察入口。

### 渠道扩展不等于能力升级

多接一个渠道只是多了一个入口，不代表系统本身更稳定、更安全。

### 模型配置不等于全部配置

模型只是能力层的一部分，真正影响稳定性的往往是工具、权限、入口和认证。
