# 渠道能力概览

OpenClaw 官方文档把 Channels 单独作为一个大类，这本身就说明：多渠道能力不是附属功能，而是系统结构的核心部分。它的重点不在于“支持多少个平台”，而在于“同一套代理能力怎样被不同入口稳定使用”。

## 为什么渠道能力重要

普通聊天产品通常只有一个主入口，而 OpenClaw 更强调把同一个助手体系接到不同入口上。

这带来的变化是：

- 你不需要在每个平台重新建立一套助手
- 不同入口可以共享更一致的能力结构
- 后续扩展工具、模型和规则时，不必每次都从零开始

## 更适合怎么理解渠道

不要一开始就把 Channels 理解成“支持了多少个平台”，更好的理解方式是：

- 它是消息入口层
- 它连接的是同一个助手系统
- 它的价值在于统一，而不只是数量

## 常见渠道类型可以怎么分

从使用场景看，常见入口大致可以分成三类：

### 1. 个人高频入口

适合你每天都在用的消息或工作入口。它的重点是响应快、进入成本低。

### 2. 团队协作入口

适合多人共享一个代理或让助手嵌入团队工作流。它的重点是权限、通知和可追踪性。

### 3. 管理与观察入口

这类入口不是为了高频对话，而是为了配置、诊断和审计，例如 [Control UI](/docs/control-ui)。

## 中文用户做选择时该看什么

### 1. 先看自己主要在哪里使用

如果你主要通过某类消息工具工作，就优先看那个渠道，不要同时展开多个入口。

### 2. 再看维护成本

多渠道不是越多越好。对第一次使用来说，先跑通一条渠道，通常比同时接三条更稳。

### 3. 最后看安全和边界

每多一个渠道，就多一层入口和风险面。所以渠道扩展应当放在已经理解基础结构之后再做。

### 4. 评估这个入口是否真的需要长期维护

很多人第一次接入时会把“看起来方便”的入口都接上，结果后面每次升级都要逐个检查。更稳的做法是只保留真正高频、真正需要的入口。

## 推荐的第一次接入顺序

1. 一个管理入口
2. 一个最常用的消息入口
3. 跑通最小链路
4. 再决定是否扩展第二个渠道

## 多渠道接入时最容易出的问题

### 同一个系统被当成多个独立系统来配

渠道只是入口，真正的代理、模型和工具策略应尽量统一管理。

### 新增渠道前没有先验证认证边界

入口一多，凭证、回调和访问控制就容易混乱。先检查认证，再开放入口。

### 渠道问题和系统问题混在一起排

如果某个入口不可用，先回到 Control UI 和日志层确认系统是否健康，再判断是不是单个渠道本身的问题。
