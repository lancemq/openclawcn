# ClawHub 的搜索、信号和信任边界

前一页已经解释了 ClawHub 的版本、锁文件和 `sync`。  
但如果你真的开始长期用技能生态，更常见的问题其实是另一类：

- 这么多技能怎么找
- 哪些技能值得先试
- 公共注册表里的“热度”到底能不能信

最新官方 ClawHub 文档把这些问题讲得比以前更完整了。

## ClawHub 现在不只是下载入口

根据当前官方文档，ClawHub 至少同时承担三层作用：

- 公共技能注册表
- 搜索与发现界面
- 公开 moderation 平台

这说明它已经不是“传文件的地方”，而是一个带排序、举报和可见性治理的公共生态面。

## 搜索现在不只是关键词匹配

官方文档现在明确提到：

- ClawHub 的搜索由 embeddings / vector search 驱动

这意味着你搜索技能时，不只是名字完全对上才会出现，而更像是在找“语义上接近”的能力包。

对中文用户来说，这点很重要，因为你在搜索时更适合用：

- 场景词
- 任务词
- 工具词

而不只是赌自己能猜中作者的精确命名。

## 哪些信号值得看

官方文档当前明确提到，ClawHub 会记录一些 usage signals，例如：

- stars
- downloads

这些信号的意义不是“高热度一定安全”，而是帮助排序和发现。

更稳的理解方式是：

- 热度信号可以帮助你缩小候选范围
- 但不能代替对 `SKILL.md` 和文件内容的检查

## 最值得先看的其实是 bundle 内容

因为 ClawHub 存的是 versioned skill bundle，所以你能看到的不是一个标题，而是一整份技能目录内容。

真正值得优先检查的通常是：

- `SKILL.md` 是否把能力边界写清楚
- 是否要求额外环境变量或 API key
- 是否依赖外部脚本或额外下载
- 是否声明了明显的副作用

这比只看“这个技能是不是热门”更有价值。

## 举报和 moderation 机制在解决什么问题

ClawHub 是公开生态，所以官方没有把它包装成“默认可信市场”，而是明确提供了治理机制。

当前官方文档里比较关键的点包括：

- 任何登录用户都可以举报
- 举报原因必须填写
- 单个用户的活跃举报数有限制
- 一个技能被多个不同用户举报后会自动隐藏
- moderator 可以查看、恢复、删除或封禁

这说明官方默认承认：

- 公开技能生态会出现低质量或有风险内容
- 需要靠 moderation 维持基本可用性

## 为什么“公开”本身就意味着要保守使用

官方文档对 ClawHub 的定位很直接：

- 所有技能默认公开
- 任何人都能浏览和分享

这意味着你更应该把它理解成：

- 开源生态入口

而不是：

- 官方逐个审核过的能力商店

所以更稳的做法不是“看到能装就装”，而是：

1. 先看描述和标签
2. 再看 bundle 内容
3. 再决定要不要装进当前 workspace

## 中文用户怎么用这些信号会更稳

一个很实用的判断顺序通常是：

### 先看是否真的贴合当前任务

和当前项目无关的技能，即使再热门，也只会增加复杂度。

### 再看是否要求高权限或秘密注入

如果技能需要：

- API key
- 本地脚本执行
- 宿主机环境变量

那就应该用更高的警惕级别去看它。

### 最后再看 stars / downloads

热度可以帮助你减少试错，但不应该反过来替你做安全判断。

## 什么样的技能更适合先试

更适合第一批尝试的通常是：

- 检索类
- 文档整理类
- 轻量数据处理类
- 明显只读的辅助技能

更不适合直接装进长期环境的通常是：

- 高权限终端执行类
- 涉及生产写入的自动化类
- 会修改长期记忆或会话状态的能力包

## 为什么搜索和治理要一起理解

ClawHub 现在已经把“发现”和“治理”放在一个系统里：

- 搜索帮你找到东西
- 举报和隐藏帮你过滤风险
- tags / signals 帮你排序

这三件事一起看，才能真正理解公共技能注册表怎么用，而不是只会 `install`。

## 一条更稳的使用路径

建议按这个顺序来：

1. 用自然语言搜索缩小范围
2. 看 `SKILL.md` 和 bundle 内容
3. 优先在独立 workspace 安装
4. 用少量高频技能开始，而不是一次装很多
5. 长期稳定后再考虑把常用技能同步或发布到自己的节奏里
