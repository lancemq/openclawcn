# 社区支持

对于持续演进的项目来说，社区支持不是附属页，而是文档体系的一部分。很多信息并不会第一时间写进正式文档，而是先出现在 issue、release、讨论区或用户分享里。对中文用户来说，建立正确的提问和跟踪路径，能显著降低信息断层。

## 优先关注的官方入口

- GitHub 仓库：适合查看源码、release、issue 和讨论上下文
- Releases：适合跟踪版本变化和迁移提醒
- 官方文档：适合理解当前推荐用法和结构调整
- 站内反馈页：适合把中文用户的阅读和使用问题整理成可处理的问题单

## 遇到问题时应该去哪里

### 适合先看文档的情况

- 你还在理解概念
- 你还没完成 onboarding
- 你不确定问题属于安装、配置还是使用方式

### 适合提 issue 的情况

- 你已经能稳定复现问题
- 你知道大致发生在什么模块
- 你能提供版本、平台、日志和复现步骤

### 适合先在社区讨论的情况

- 你想比较不同配置方案
- 你不确定是否属于 bug
- 你更需要经验建议，而不是立刻修复

## 提问前建议准备的最小信息

无论你在什么渠道提问，最好都准备好这些内容：

1. 当前使用的 OpenClaw 版本
2. 操作系统和运行环境
3. 是否经过 onboarding、是否能打开 Control UI
4. 是否只影响单个渠道，还是系统整体不可用
5. 能稳定复现的步骤
6. 关键报错和日志片段

## 更高效的问题描述方式

推荐按下面的顺序组织问题：

1. 你想做什么
2. 你目前怎么配置的
3. 实际发生了什么
4. 你已经尝试过哪些排查动作

这样别人更容易判断这是安装问题、渠道问题、模型问题还是安全边界问题。

## 贡献内容时从哪里开始

如果你还不准备改代码，也完全可以从下面几类贡献开始：

- 补充中文使用说明
- 为版本更新写中文摘要
- 把常见问题整理成 FAQ
- 提交更清晰的复现步骤和截图

## 中文用户的参与建议

- 阅读官方 release 后，把关键变化翻译成团队内部可执行的提醒
- 遇到不确定的问题，先查 [故障排除与诊断思路](/docs/reference/troubleshooting)
- 如果涉及版本变化，先看 [版本迁移与升级检查单](/docs/setup/migration-guide)
