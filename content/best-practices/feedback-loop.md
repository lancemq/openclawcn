# 反馈闭环设计

如果用户不知道应该去哪里反馈问题，很多信息就会停留在私人聊天、零散评论或未整理的截图里。

## 建议路径

1. 先用 FAQ 和搜索自助查找
2. 仍然找不到时提交站内反馈，说明缺了什么信息
3. 对源码、功能和长期讨论问题再进入 GitHub Issues 或 Discussions

## 为什么这样更合适

- 用户不需要一上来就判断“我该不该提 Issue”
- 站内可以先沉淀常见中文问题
- GitHub 更适合承接需要公开追踪的问题

## 推荐做法

- FAQ 解决重复问题
- 反馈表单收集内容缺口
- GitHub 承接源码和长期协作

这样比所有问题都直接扔进一个入口更稳定。
