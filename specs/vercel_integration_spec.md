# Vercel 内部部署规格说明

## 1. 模块概述
该规格用于说明 OpenClawCN 项目如何部署到 Vercel，以及开发时需要遵守哪些平台约束。它属于项目内部技术文档，不属于对外网站内容模块。

## 2. 目标用户
- 希望快速部署 OpenClaw 演示站的开发者
- 需要 Preview 环境进行内容审核的维护者
- 希望通过 GitHub + Vercel 建立自动发布流程的团队

## 3. 功能需求

### 3.1 内部部署文档
- **部署前准备**：GitHub 仓库、Node 版本、构建命令、输出目录
- **一键部署流程**：导入模板、创建项目、确认构建配置
- **环境变量说明**：区分 Development / Preview / Production
- **域名绑定**：自定义域名、HTTPS、回源和跳转策略
- **发布与回滚**：查看部署记录、回滚版本、排查构建失败
- **Webhook/Hook**：内容更新后触发重新部署
- **运行时约束**：无状态函数、只读文件系统、外部数据库和对象存储

### 3.2 展示内容
- Vercel 集成流程图
- 常见环境变量列表示例
- Preview URL 用例说明
- 国内访问提示：何时使用 Vercel，何时同步到国内 CDN

## 5. 技术要求
- 推荐技术栈为 Nuxt 3 + Nitro + TypeScript，默认构建命令为 `nuxt build`
- 所有服务端能力默认运行在 Vercel Functions 的 Node.js Runtime
- 不依赖本地持久化磁盘；上传和媒体文件统一使用对象存储
- 对接 GitHub 仓库触发自动部署
- 保留 Deploy Hook、Analytics、Speed Insights、Observability 的扩展位
- 如后续拆分独立后台，仅允许采用 Vercel 官方支持的 Node.js 框架与入口约定

## 5.1 技术栈约束
- **前端**：必须使用 Vercel 有一线支持的框架能力，优先选择 Nuxt 3
- **后端**：优先使用 Nuxt Server API；避免默认采用需要长驻进程假设的独立服务
- **数据层**：数据库、缓存、文件存储全部外部托管，通过环境变量接入
- **任务处理**：长任务、定时任务、异步处理必须基于 Vercel Functions / Cron 重新设计

## 6. 文案与设计要求
- 风格应偏工程化、可信、简洁，避免营销化空话
- 该内容应放在内部部署文档中，而非首页和对外文档导航中
- 使用流程卡片、环境变量表格、状态标签来降低理解成本
- 明确标注 Preview、Production、Rollback 等术语的中文解释

## 7. 验收标准
- [ ] 对外网站页面不出现独立的 Vercel 对接入口
- [ ] 项目内部文档能找到完整的 Vercel 部署教程
- [ ] 至少覆盖模板部署、环境变量、域名绑定、回滚四类内容
- [ ] 用户可在 10 分钟内完成一次从 GitHub 到 Vercel 的部署
